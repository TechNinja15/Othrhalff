DATABASE FETCHING PERFORMANCE ISSUES

1. AppLayout.tsx (Exact Counts on Large Tables)
   - Issue: The layout fetches exact counts for match badges using `supabase.from('matches').select('*', { count: 'exact', head: true }).or(...)`.
   - Impact: As the `matches` table grows, exact counts using `OR` conditions table-scan heavily. This significantly slows down the initial shell/layout load of the application.
   - Fix: Use a dedicated tracking/summary table, PostgreSQL triggers to maintain a running count, or use `estimated` counts instead of `exact`.

2. Matches.tsx (Over-fetching on Realtime Subscriptions)
   - Issue: Whenever a new message arrives via the realtime subscription on the `messages` table, the component calls `loadMatches(true)` which executes the heavy `get_matches_with_preview` RPC. 
   - Impact: If 5 messages arrive quickly in succession, it slams the database 5 times unnecessarily to fetch data it mostly already has.
   - Fix: Optimistically append the new realtime message data directly into the local `chats` state array instead of re-running the heavy database RPC on every ping.

3. Confessions.tsx (N+1 Queries & Payload Size)
   - Issue: The main feed fetches confessions and then fetches associated relationships. While joins are used, it pulls a lot of nested data. Additionally, it fetches the user's specific poll votes in a separate query (`myVotes`) after fetching the confessions, creating an N+1 style secondary lookup.
   - Impact: Increases database round-trips and memory overhead. When fetching the feed, it fetches comments inline which can bloat the payload if a confession goes viral.
   - Fix: 
     a) Optimize the bulk `select` payload by only requesting strictly necessary columns from relational joins.
     b) Use a Postgres function (RPC) to fetch confessions with user votes combined, eliminating the secondary `poll_votes` query string.
     c) Strictly enforce pagination/limits on the nested comments fetch.

4. General Realtime Overhead
   - Issue: Multiple components (`AppLayout`, `Matches`, `Chat`, `Confessions`) establish their own Supabase realtime channels.
   - Impact: Too many concurrent websocket connections or heavy payloads over the websocket can cause memory leaks or sluggishness on low-end mobile devices.
   - Fix: Consolidate realtime listeners where possible, or ensure channels are strictly filtered and properly cleaned up on unmount.
